## 逻辑数据模型设计规范

### 1. 概述

#### 1.1 规范定位

逻辑数据模型是概念模型的细化，面向**数据设计人员**，包含**逻辑实体、属性、主外键、关系**，独立于具体数据库实现。它是物理模型设计的直接蓝图。

#### 1.2 适用角色

- 数据架构师（主责）
- 数据库设计工程师
- 后端开发工程师（评审）

#### 1.3 输入

- 业务概念模型
- 第一阶段业务模型设计规范中的“属性定义表”、“实体关系定义”
- 数据标准库（已有属性标准）

#### 1.4 输出

- 逻辑实体定义表
- 属性定义表（含数据类型、长度、约束）
- 逻辑ER图（使用Mermaid erDiagram或数据库建模工具）
- 逻辑模型文档

### 2. 核心要素

| 要素         | 定义                                       | 示例                  |
| :----------- | :----------------------------------------- | :-------------------- |
| **逻辑实体** | 对应业务对象或子对象，具有唯一标识（主键） | DigitalBusinessCard   |
| **属性**     | 描述实体的特征，需定义业务数据类型         | status (枚举)         |
| **主键**     | 唯一标识实体的属性或组合                   | card_id               |
| **外键**     | 引用其他实体的主键，表示关系               | employee_id           |
| **关系**     | 1:1, 1:N, N:N                              | 名片对访客记录（1:N） |

### 3. 设计原则

- **满足第三范式（3NF）**：每个属性依赖于完整主键，不存在传递依赖。但可根据查询性能适当反范式（如冗余常用字段）。
- **命名规范**：
  - 逻辑实体名使用帕斯卡命名（PascalCase），如 `DigitalBusinessCard`。
  - 属性名使用驼峰命名（camelCase），如 `cardId`。
  - 外键与目标实体的主键同名。
- **主键设计**：建议使用无业务含义的代理键（如雪花ID、UUID），不依赖业务字段。
- **数据分类**：每个逻辑实体需标记数据分类（主数据/事务数据等），用于指导物理存储策略。
- **审计字段**：每个逻辑实体必须包含 `createTime`、`updateTime`、`createBy`、`isDeleted`。

### 4. 逻辑实体定义模板

| 逻辑实体名          | 实体描述 | 数据分类 | 主键     | 外键                         |
| :------------------ | :------- | :------- | :------- | :--------------------------- |
| DigitalBusinessCard | 数字名片 | 主数据   | cardId   | employeeId → Employee        |
| VisitRecord         | 访客记录 | 观测数据 | recordId | cardId → DigitalBusinessCard |
| Lead                | 线索     | 事务数据 | leadId   | cardId, ownerEmployeeId      |

### 5. 属性定义模板

| 实体                | 属性名    | 业务数据类型 | 长度 | 必填 | 业务约束                  | 关联数据标准 |
| :------------------ | :-------- | :----------- | :--- | :--- | :------------------------ | :----------- |
| DigitalBusinessCard | cardId    | 字符串       | 32   | 是   | 全局唯一                  | -            |
| DigitalBusinessCard | status    | 枚举         | -    | 是   | 草稿/已发布/已禁用/已归档 | 名片状态标准 |
| DigitalBusinessCard | qrCodeUrl | 字符串       | 255  | 否   | URL格式                   | -            |

### 6. 逻辑ER图规范

使用Mermaid的 `erDiagram` 语法，标注关系类型。

```mermaid
erDiagram
    Employee ||--|| DigitalBusinessCard : has
    DigitalBusinessCard ||--o{ VisitRecord : generates
    VisitRecord ||--|| Lead : generates
    Lead }o--|| Employee : assigned_to
```



### 7. 输出示例（企业数字名片片段）

**逻辑实体**：DigitalBusinessCard

| 属性       | 数据类型 | 长度 | 必填 | 说明            |
| :--------- | :------- | :--- | :--- | :-------------- |
| cardId     | VARCHAR  | 32   | 是   | 名片ID，主键    |
| employeeId | VARCHAR  | 32   | 是   | 员工ID，外键    |
| status     | VARCHAR  | 20   | 是   | 状态，默认DRAFT |
| qrCodeUrl  | VARCHAR  | 255  | 否   | 名片码URL       |
