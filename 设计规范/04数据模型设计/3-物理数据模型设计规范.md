## 物理数据模型设计规范

### 1. 概述

#### 1.1 规范定位

物理数据模型是逻辑模型在**特定数据库管理系统（DBMS）**上的实现，描述**表、字段、索引、分区、存储参数**等物理细节，面向DBA和开发人员。

#### 1.2 适用角色

- DBA（数据库管理员）
- 后端开发工程师
- 运维工程师

#### 1.3 输入

- 逻辑数据模型
- 数据库选型（MySQL、PostgreSQL等，MySQL默认支持MySQL 5.6+）
- 容量规划（数据量、QPS）

#### 1.4 输出

- 建表DDL脚本（含注释）
- 索引设计文档
- 分区方案（如需）
- 物理数据模型图（工具导出）

### 2. 核心内容

| 内容       | 说明                                                         |
| :--------- | :----------------------------------------------------------- |
| **物理表** | 每个逻辑实体对应一张或多张表（如水平分表）                   |
| **字段**   | 指定具体DBMS数据类型（如MySQL的 `VARCHAR(64)`、`DECIMAL(18,2)`）、默认值、是否为空 |
| **索引**   | 主键索引、唯一索引、普通索引、复合索引                       |


### 3. 设计原则

- **表命名**：`{业务域}_{逻辑实体名}`，如 `crm_digital_business_card`。
- **字段命名**：使用下划线分隔，如 `card_id`、`create_time`。
- **审计字段**：强制包含 `create_time`（DATETIME）、`update_time`（DATETIME，自动更新）、`is_deleted`（TINYINT，逻辑删除）、`create_by`（VARCHAR）。
- **字符集**：MySQL统一使用 `utf8mb4`，排序规则 `utf8mb4_general_ci`。
- **存储引擎**：MySQL使用InnoDB。
- **注释**：表和字段必须有注释，业务含义清晰。

### 4. 索引设计规范

| 索引类型     | 规范                                                         |
| :----------- | :----------------------------------------------------------- |
| **主键索引** | 使用自增ID（BIGINT）或雪花ID，避免UUID作为主键（碎片化）     |

### 5.DDL建表要求

每张表中需要将以下字段必要字段，且必须包含，且字段类型和下面保持一致：

```sql
`id` varchar(32)  NOT NULL COMMENT '主键id',
`created_by` varchar(80) COMMENT '创建人',
`updated_by` varchar(80) COMMENT '更新人',
`created_at` datetime NULL DEFAULT NULL COMMENT '创建时间',
`updated_at` datetime NULL DEFAULT NULL COMMENT '更新时间',
```


### 6. DDL脚本示例（MySQL）

```sql
sql
CREATE TABLE t_digital_business_card (
    id          VARCHAR(32)   NOT NULL COMMENT '名片ID（雪花ID）',
    employee_id      VARCHAR(32)   NOT NULL COMMENT '员工ID',
    status           VARCHAR(20)   NOT NULL DEFAULT 'DRAFT' COMMENT '状态：DRAFT/PUBLISHED/DISABLED/ARCHIVED',
    qr_code_url      VARCHAR(255)  NULL COMMENT '名片码URL',
    create_time      DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time      DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    create_by        VARCHAR(50)   NOT NULL COMMENT '创建人',
    is_deleted       TINYINT       NOT NULL DEFAULT 0 COMMENT '逻辑删除：0-未删除，1-已删除',
    PRIMARY KEY (card_id),
    INDEX idx_employee (employee_id),
    INDEX idx_status (status),
    INDEX idx_create_time (create_time)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='数字名片表';
```
