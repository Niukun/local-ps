# Bug 修复工作流

## 何时使用

- 用户报告软件行为不符合预期（界面显示异常、功能不可用、报错等）
- 用户提出"修复 xx 问题"、"xx 不生效"、"xx 有 Bug"等诉求
- Code Review 中发现潜在缺陷需要修复

---

## 工作流程（严格按顺序执行）

### 步骤 1：问题理解与澄清

充分理解用户描述的问题。如有以下情况，**必须先向用户提问澄清，确认后才能继续**：

- 用户描述的现象模糊、不完整（如"页面报错"但没有截图或错误信息）
- 复现步骤缺失，无法确定操作路径
- 涉及多人协作或跨模块行为，不确定影响范围
- 用户使用了不确定性的措辞（"好像"、"大概"、"可能是"）

> 原则：宁可多问一句，也不要猜错需求。

### 步骤 2：Code Review 与根因分析

在问题明确后：

1. **定位相关代码**：使用 grep/glob/read 等工具找到问题涉及的代码文件
2. **仔细阅读代码**：理解代码逻辑、数据流、边界条件
3. **寻找根因**：不要只看表象，追查问题的源头
   - 数据源是否正确？
   - 类型/空值处理是否完备？
   - 异步时序是否存在竞态？
   - 边界条件是否覆盖？
   - 是否有隐含的状态依赖？
4. **必要时自行编写测试用例验证**，确认对问题的理解

### 步骤 3：方案输出与确认

**在修改任何代码之前**，必须向用户输出以下内容：

1. **问题根因**：用简明的语言说明问题出在哪里
2. **代码错误位置**：具体到文件路径和行号
3. **修复方案**：提出 1 个或多个可行方案（如有多个方案，说明各自的优缺点）

然后**等待用户确认**，或根据用户反馈调整方案。

> 禁止跳过此步骤直接改代码！

### 步骤 4：执行修复

用户确认方案后：

1. 执行代码修改
2. 修改后运行项目相关测试/构建验证改动无误
3. 如涉及用户界面变化，可截图/录屏供用户确认

### 步骤 5：禁止提交

**修改完成后禁止执行 git commit / git add / git push 等操作**，除非用户明确要求。
